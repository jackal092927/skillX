# SkillX Refine Bundle Contract v0.1

- **Project:** `projects/multi-agent-protocol-design`
- **Date:** 2026-03-23
- **Role:** input/output contract for the multi-round `C4` refine loop
- **Status:** planning draft

---

## 1. Why this document exists

`skillx-refine-protocol-v0.1.md` defines **how** `C4` should work:
- fixed protocol,
- multi-round loop,
- tune/eval separation,
- bounded edit surface,
- round-level memory,
- held-out evaluation only at the end.

This document defines something narrower and more operational:

> **What exact bundle of files/artifacts should a `C4` refiner read, and what exact outputs should it write each round?**

Without this contract, a multi-round refine loop is too underspecified to run reliably across machines, agents, or sessions.

---

## 2. Core principle

For `C4 v0.1`, the refiner should be:

> **local-evidence-first, contrastive, bounded, and auditable**

That means:
- local execution evidence is primary,
- the refiner should see both failure and success/near-success signals,
- the allowed inputs are limited,
- and every round should leave durable artifacts.

The refiner should **not** rely on arbitrary open-ended search during the loop.

---

## 3. Contract layers

The refine bundle has three layers.

### 3.1 Static bundle
Inputs that should stay stable across the entire loop.

### 3.2 Round bundle
Inputs/outputs specific to each refine round.

### 3.3 Final bundle
Artifacts needed to choose `C4-final` and evaluate it on held-out tasks.

---

## 4. Static bundle — required inputs

These files must exist before the first refine round starts.

### 4.1 Protocol reference
- `plans/skillx/skillx-refine-protocol-v0.1.md`
- this contract file itself

These define the rules of the loop.

### 4.2 Skill ancestry
Required:
- current `C3` skill artifact
- original `C1` skill snapshot

Strongly recommended:
- `C2 -> C3` diff summary
- rewrite notes from earlier authoring stages

Purpose:
- keep lineage explicit
- avoid losing the original task intent
- prevent accidental strategy drift

### 4.3 Task context
Required:
- task instruction
- task metadata / difficulty / category
- task-level environment notes if relevant

Purpose:
- ground refine in the real benchmark task rather than abstract skill prose alone

### 4.4 Tune evidence summary
Required:
- condition result summary for the task or task family
- deterministic verifier outputs
- at least one failure or weak case
- at least one success or near-success case for contrast

Purpose:
- make refine evidence-grounded instead of purely speculative

### 4.5 Refine constraints
Required:
- allowed edit surface
- forbidden edit surface
- round budget
- stopping criteria
- explicit ban on held-out eval visibility

Purpose:
- keep `C4` bounded and interpretable

---

## 5. Static bundle — strongly recommended inputs

These are not strictly required, but in most serious runs they should be present.

### 5.1 Negative-transfer note
Record cases where the skill appears to hurt.

Why:
- average gain is not the only story
- SkillX explicitly cares about conditional utility and harm control

### 5.2 Suspicious-success note
Record cases where a pass may be shallow, exploitative, or misaligned.

Why:
- otherwise the refiner may learn to optimize evaluator blind spots

### 5.3 Manual audit note
A human or independent evaluator summary of important non-deterministic issues.

Why:
- deterministic verifiers often miss scope violations, misleading strategies, or shortcut behavior

### 5.4 Authoring burden note
Record whether the skill is becoming significantly heavier, harder to maintain, or less adoption-friendly.

Why:
- SkillX is supposed to stay lightweight by default
- cost and complexity are part of the result

### 5.5 Cross-task pattern memo
For Pilot B or later, a tune-split-only memo summarizing repeated failure patterns across multiple tasks.

Why:
- multi-round refine should exploit accumulated patterns, not only per-task local noise

---

## 6. Static bundle — optional inputs

These are allowed, but should be treated as optional and controlled.

### 6.1 Frozen heuristic library
A small pre-approved note of refine heuristics.

Examples:
- tighten `scope_out` when over-activation is common
- strengthen `requires` when missing prerequisites repeatedly cause damage
- compress verbose body guidance if the agent appears not to use it

### 6.2 Prior refine memos from other tasks
Useful only if they are clearly comparable and do not smuggle in irrelevant task-specific hacks.

### 6.3 Curated research memo
A human-prepared short note containing high-level lessons from prior literature or internal research.

Important:
- this should be pre-curated
- not generated by arbitrary mid-loop web search

---

## 7. Forbidden inputs

The following should **not** be present in the bundle for `C4 v0.1`:

### 7.1 Held-out eval traces or cases
Hard prohibition.

### 7.2 Hidden test details or oracle solutions
Hard prohibition.

### 7.3 Arbitrary open-ended internet search results during the loop
Default prohibition.

Reason:
- this changes the problem from bounded refine to free research
- it muddies attribution and increases benchmark contamination risk

### 7.4 Unfiltered unrelated chat history
Do not dump whole conversations into the bundle.
Only distilled, task-relevant artifacts should be used.

### 7.5 Free protocol mutation instructions
The bundle should not instruct the refiner to redesign the protocol mid-loop.

---

## 8. Minimum contrastive evidence requirement

A valid bundle should include **contrastive evidence**, not only failures.

### 8.1 Required contrast
At minimum, the refiner should be able to compare:
- one failure / weak case
- one success / near-success case

### 8.2 Why this matters
If the refiner only sees failure, it may over-tighten the skill and collapse its usefulness.

Contrastive evidence helps it learn:
- what to preserve
- not only what to remove

---

## 9. Round bundle — per-round required inputs

Every round `Ri` should read a round-specific bundle.

### 9.1 Required per-round inputs
- current skill candidate (`R{i-1}` or `C3` for round 1)
- refine ledger up to the previous round
- latest tune-side result summary
- diagnosis table from previous round or current update
- active constraints

### 9.2 Strongly recommended per-round inputs
- previous round refine memo
- previous round diff summary
- previous round risk note
- any newly added suspicious-success or negative-transfer observations

### 9.3 Per-round rule
A later round must read the durable outputs of earlier rounds.

This is the minimum persistence requirement that makes the loop genuinely multi-round rather than a sequence of unrelated one-shot edits.

---

## 10. Per-round required outputs

Each round should emit a standard artifact set.

### 10.1 Required outputs
For round `Ri`, write:

1. `round_i_skill.md`
2. `round_i_refine_memo.md`
3. `round_i_diff_summary.md`
4. `round_i_effect_estimate.md`
5. `round_i_risk_note.md`
6. `round_i_diagnosis_table.md`

### 10.2 Required content of each

#### `round_i_skill.md`
- the actual refined skill candidate for this round

#### `round_i_refine_memo.md`
Must include:
- what changed
- why it changed
- which evidence justified it
- what uncertainty remains

#### `round_i_diff_summary.md`
Must include:
- field-level or section-level summary of edits from the previous round

#### `round_i_effect_estimate.md`
Must include:
- predicted gain
- predicted harm risk
- confidence level
- whether this round mainly tightened / clarified / repaired / compressed

#### `round_i_risk_note.md`
Must include:
- what new regressions this round might introduce
- especially over-tightening, loss of generality, or increased complexity

#### `round_i_diagnosis_table.md`
Must include at least these columns:
- issue
- evidence
- suspected cause
- edit target
- confidence

## 11. Refine ledger contract

The multi-round loop requires one cumulative memory object.

### 11.1 Required ledger file
Recommended canonical path:
- `refine_ledger.md`

### 11.2 Ledger must track
For each round:
- round number
- parent candidate
- key edits made
- evidence used
- predicted effect
- observed tune-side reaction if available
- whether the same type of edit is repeating
- whether the loop appears to be converging or drifting

### 11.3 Why the ledger is mandatory
Without a ledger:
- the loop loses memory,
- repeated edits are harder to detect,
- and later diagnosis becomes much less trustworthy.

---

## 12. Bundle manifest contract

Every refine run should have a machine-readable manifest.

### 12.1 Suggested file
- `bundle_manifest.json`

### 12.2 Required top-level fields
Suggested structure:

```json
{
  "experiment_id": "skillx-skillsbench-001",
  "run_id": "pilot-b-refine-001",
  "protocol_version": "skillx-refine-protocol-v0.1",
  "bundle_contract_version": "skillx-refine-bundle-contract-v0.1",
  "starting_artifact": "C3",
  "round_budget": 4,
  "heldout_eval_visible": false,
  "static_inputs": {},
  "rounds": [],
  "final_selection": {}
}
```

### 12.3 Purpose
This makes the refine run portable and auditable across machines and agents.

---

## 13. Final bundle — required outputs

At the end of the loop, the refine run should produce a final bundle.

### 13.1 Required final outputs
1. `C4-final/SKILL.md`
2. `C4-final/final_refine_memo.md`
3. `C4-final/final_diff_summary.md`
4. `C4-final/final_selection_note.md`
5. `C4-final/heldout_eval_report.md`

### 13.2 Selection note must answer
- why this round was chosen over the other candidate rounds
- whether later rounds actually improved the artifact
- whether an earlier round was selected instead of the last one

### 13.3 Held-out evaluation report
This is the only valid place to make the final `C4` performance claim.

---

## 14. Selection rule for `C4-final`

`C4-final` should not automatically be the last round.

### 14.1 Candidate pool
The selector may compare:
- `R0` (`C3`)
- `R1`
- `R2`
- `R3`
- ...

### 14.2 Recommended decision criteria
Choose the candidate with the best tradeoff on tune evidence among:
- performance
- harm control
- reliability
- complexity / authoring burden

### 14.3 Conservative rule
If later rounds do not clearly dominate, selecting an earlier round is acceptable and should be explicitly recorded.

```text
experiments/skillx-skillsbench-001/
  runs/
    pilot-b/
      refine/
        bundle_contract_v0.1.md
        bundle_manifest.json
        static/
          ancestry/
          task_context/
          tune_evidence/
          constraints/
          audits/

          round-0/

            round_0_refine_memo.md
            round_0_diff_summary.md
            round_0_effect_estimate.md
            round_0_risk_note.md
            round_0_diagnosis_table.md
          round-1/
          round-2/
          round-3/
        refine_ledger.md
        C4-final/
          SKILL.md
          final_refine_memo.md
          final_diff_summary.md
          final_selection_note.md
          heldout_eval_report.md
```

The exact names may change, but the contract principles should remain:
- stable static inputs
- explicit round artifacts
- cumulative ledger
- explicit final selection bundle

---

## 16. v0.1 operational defaults

For the first practical use of this contract, the default assumptions should be:

- **protocol:** frozen
- **execution style:** multi-round
- **round budget:** 3–4 rounds
- **evidence source:** tune split only
- **evaluation claim:** held-out only
- **external search during loop:** disallowed by default
- **memory object:** mandatory ledger
- **contrastive evidence:** required

---

## 17. Bottom line

For `C4 v0.1`, Pi-AutoResearch should read a **bounded refine bundle**, not the whole world.

That bundle should be:
- local-evidence-first,
- contrastive,
- tune-only,
- memory-preserving,
- and explicit about both allowed edits and forbidden information.

This is what makes multi-round refine:
- reproducible,
- comparable,
- and scientifically interpretable.
